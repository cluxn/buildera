<title>Contrast audit + brand-primary link remediation</title>

<action>
Compute WCAG contrast ratios for the critical pairs from research using converted hex values: body text on white, muted-foreground on white and on muted bg, brand-primary text on white, btn-primary white-on-gradient, label text. Record each computed ratio and PASS/FAIL in the SUMMARY (4.5:1 body, 3:1 large-text threshold). Apply fixes ONLY where a pair fails: (a) if brand-primary (`hsl(221 83% 53%)` ≈ #2563eb) on white fails 4.5:1 for small text, ensure a `--brand-primary-dark` token (≈ `hsl(221 83% 40%)`) exists and is used for inline link/label text on white — do not change the button/gradient brand-primary; (b) if muted-foreground on muted background fails, darken the muted-foreground token enough to reach 4.5:1. Do not change any pair that already passes. List exactly which selectors/usages switch to the darker token.
</action>

- `cd buildera-frontend && npm run build` exits 0

<task id="09-03-T2">
<title>Visible focus-visible ring + 48px touch targets</title>
<read_first>
- buildera-frontend/src/app/globals.css (the global `* { outline-ring/50 }` rule and @layer base)
- buildera-frontend/src/components/sections/contact/ContactInfo.tsx (icon containers w-10 h-10 = 40px — under 48px)
- buildera-frontend/src/components/layout/SiteNavClient.tsx (nav links / CTA — verify tap sizes)
- buildera-frontend/src/components/layout/MobileNavDrawer.tsx (drawer toggle and links — verify tap sizes)
</read_first>
<action>
Add a global `:focus-visible` rule in globals.css `@layer base` that renders a clearly visible ring: `outline: 2px solid var(--brand-primary); outline-offset: 2px;` (use `:focus-visible`, NOT `:focus`, so mouse clicks don't show the ring). This supplements the faint `outline-ring/50`. Then audit interactive elements for 48px minimum touch targets: any icon-only link or button that is smaller than 48px (e.g. ContactInfo's `w-10 h-10` icon wrappers if they are clickable, mobile nav toggle, social icon links) must get `min-h-12 min-w-12` plus flex centering. Non-clickable decorative icon containers (like ContactInfo's icon badges, which are inside larger card links) do not need resizing — only resize actual tap targets. Document which elements were resized in the SUMMARY.
</action>
<acceptance_criteria>
- `grep -n ":focus-visible" buildera-frontend/src/app/globals.css` matches and uses `var(--brand-primary)`
- No use of bare `:focus {` for the new outline (must be `:focus-visible`)
- SUMMARY lists every interactive element audited for 48px and which were resized to `min-h-12 min-w-12`
- `cd buildera-frontend && npm run build` exits 0
</acceptance_criteria>
</task>

<task id="09-03-T3">
<title>Alt text + aria-hidden + keyboard navigation verification</title>
<read_first>
- buildera-frontend/src/components/layout/SiteNavClient.tsx (tab order, aria-expanded on dropdown triggers)
- buildera-frontend/src/components/layout/MobileNavDrawer.tsx (focus trap, Escape to close)
- buildera-frontend/src/components/sections/contact/ContactForm.tsx (label associations, focus order)

<action>
Audit alt text and decorative icons: confirm every `<Image>` has a non-empty `alt`; confirm decorative inline SVGs / icon glyphs that convey no information carry `aria-hidden="true"` (and that icon-only controls have an `sr-only` text label or `aria-label`). Add `aria-hidden="true"` where a decorative icon lacks it and `aria-label`/`sr-only` where an icon-only control lacks an accessible name. Then perform a keyboard-navigation pass and document results in the SUMMARY: (1) Tab order is logical on header nav, (2) mega dropdown triggers expose `aria-expanded` and open via Enter/Space, (3) mobile drawer traps focus while open and closes on Escape returning focus to the toggle, (4) contact form fields are reachable in order with associated labels, (5) any modal/popup traps focus. For each item record PASS or the fix applied. Do not restructure working components — only add missing aria attributes / labels.
</action>
